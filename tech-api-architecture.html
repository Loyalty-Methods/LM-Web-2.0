Design Philosophy

The Hardest Thing to Achieve. Our Starting Point.

We prioritize the most difficult thing for a loyalty program to achieve: real-time, synchronous for everything. That is our design philosophy. Most platforms treat real-time as an add-on or an aspiration. We treat it as the foundation.

When the architecture handles real-time from day one, everything else becomes easier. Personalization. Event-driven engagement. Partner integrations. Scale.

Integration Philosophy

Designed to Break the Black Box

ReactorCX is the only enterprise loyalty platform designed from day one to adapt to your environment and let data flow freely across your ecosystem. Acting as the loyalty brain, RCX senses customer behavior across channels, interprets it, and coordinates loyalty decisions back into the systems you already use. No trapped logic. No walled gardens. Full bidirectional flow.

  • RESTful APIs for all platform operations
  • Synchronous responses for point-of-interaction use cases
  • Async patterns available when workflows require them
  • Full API documentation with sandbox environments
  • Versioned APIs with backward compatibility

Real-Time Events

Members Know the Moment Something Happens

The platform is heavily event-driven. Anything that happens generates real-time events that you can use to engage members immediately. You achieved a streak. You got a bonus. You hit a new tier. Those events come out on the event backbone and get pushed the moment conditions are met. Congratulations screens. Instant notifications. App overlays. Members get that satisfaction right away.

  • 27+ built-in event types
  • Sub-second event publishing
  • Streaming via Kafka, webhooks, Azure EventHub
  • Push notifications, SMS, email triggered by events
  • App overlays triggered the moment conditions are met
[Event flow diagram]

Multiple Integration Modes. Simultaneously.

Real-time, near real-time, or batch. The platform adapts to how your systems operate.

REST APIs (Synchronous)

Real-time operations with full RESTful API suite. Immediate response for point-of-interaction use cases.

Event API (Publish)

Events published as they occur. Webhooks, Azure EventHub, AWS SNS. 27+ built-in event types.

Streaming Ingestion

Near real-time processing via Kafka. High-volume activity without batch delays.

Batch Ingestion (FeedXChange)

Secure file drop with automated processing. Batch files processed as individual real-time transactions.

SQL Export

Analytics and enterprise reporting. Sync activity data to your data lake.

Hybrid

Multiple modes simultaneously. Real-time for some operations, batch for others. Partners operate the way they operate.

Elastic Infrastructure

Scales Up and Down With Your Volume

On 7-Eleven Day, transaction volume spikes to multiples of normal load. The platform handles it without intervention. Auto-scale kicks in, capacity expands, and when the spike passes, it scales back down. This is not an SLA target. It is operational history. Major promotional events, peak holiday periods, flash sales. The infrastructure responds automatically.

  • AWS-native auto-scaling
  • Multi-AZ redundancy
  • Containerized services with rolling upgrades
  • Peak volume handling without degradation
  • Scale events logged and reportable

Operational Continuity

Configuration Changes Without Restarts

The platform publishes configuration changes without platform restarts. New promotions go live. Rules update. Tier thresholds adjust. All without taking the system offline. This is not just about deployment patterns. It is about operational continuity for programs that cannot afford interruption.

  • Hot configuration publishing
  • Rolling service upgrades
  • Feature flags for controlled rollout
  • Rollback at service level
  • Zero planned downtime windows
[Deployment flow diagram]

Every Transaction. Same Path. Same Speed.

Whether a transaction arrives via API, streaming, or batch file, it processes through the same real-time engine. FeedXChange rips batch files into individual transactions. Each one evaluates against rules, triggers events, and updates balances in real time. Consistent behavior regardless of ingestion method. Consistent outcomes. Consistent logging.
